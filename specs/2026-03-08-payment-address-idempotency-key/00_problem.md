# Payment Address Idempotency Key - Problem & Goals

## Context

- Background:
  - `POST /v1/chains/{chain}/payment-addresses` currently creates a new allocation whenever the same request is replayed.
  - The correct retry-control mechanism for this API is an `Idempotency-Key` request header, not `customerReference`.
  - `customerReference` should remain merchant business data for reconciliation/display.
- Users or stakeholders:
  - Merchant backend teams retrying payment-address creation after network failures or ambiguous timeouts.
  - Payrune backend maintainers who need deterministic retry behavior.
- Why now:
  - The user clarified that duplicate protection should be driven by an idempotency header.

## Constraints (optional)

- Technical constraints:
  - Keep existing Go layout and clean architecture boundaries.
  - Keep the current allocation issuance transaction flow.
  - Keep transport-level idempotency state independent from `address_policy_allocations`.
  - Avoid turning `customerReference` into an infrastructure control field.
- Timeline/cost constraints:
  - Prefer the smallest change that adds correct idempotent replay behavior.
- Compliance/security constraints:
  - No auth or secret-handling changes.

## Problem statement

- Current pain:
  - Retrying the same allocation request can issue multiple payment addresses.
  - The previous customer-reference-based interpretation was semantically wrong for API idempotency.
- Evidence or examples:
  - The create endpoint is a write operation where request replay semantics belong to `Idempotency-Key`.
  - Merchant `customerReference` may be user-facing or reconciliation-oriented, not guaranteed to be a transport-level replay key.

## Goals

- G1:
  - Support `Idempotency-Key` header on `POST /v1/chains/{chain}/payment-addresses`.
- G2:
  - Replaying the same key with the same request payload returns the existing issued allocation.
- G3:
  - Reusing the same key with a different request payload returns deterministic `409 Conflict`.
- G4:
  - Keep concurrent same-key retries safe under DB-backed uniqueness guarantees.
- G5:
  - Preserve current non-idempotent behavior when the header is absent.
- G6:
  - Keep `customerReference` as business payload only, not as the dedupe key.
- G7:
  - Keep idempotency persistence fully independent from allocation-row storage.
- G8:
  - Do not leave an idempotency key stuck when issuance fails before an address is issued.

## Non-goals (out of scope)

- NG1:
  - Generic idempotency support for every API endpoint.
- NG2:
  - Merchant-scoped idempotency namespaces beyond the current chain-scoped API.
- NG3:
  - New retrieval endpoints by idempotency key.

## Assumptions

- A1:
  - An `Idempotency-Key` is generated by the client per logical payment-address creation attempt.
- A2:
  - Request payload equivalence for this endpoint is defined by `addressPolicyId`, `expectedAmountMinor`, and `customerReference`.
- A3:
  - Same key with different payload is a client error and should not create a second allocation.
- A4:
  - This iteration replays successful issuance results only; failed issuance attempts remain retryable with the same key.

## Open questions

- Q1:
  - Should a later iteration persist a canonical request hash instead of explicit request fields as the payload shape grows?
- Q2:
  - Should idempotency keys eventually be mandatory for public create APIs?

## Success metrics

- Metric:
  - Duplicate issued-allocation rate for same `chain + Idempotency-Key`.
- Target:
  - `0` duplicate issued allocations in sequential and concurrent retry tests.
- Metric:
  - Deterministic replay behavior for same-key same-payload requests.
- Target:
  - Replayed requests return the same `paymentAddressId` and `address`.
- Metric:
  - Deterministic conflict behavior for same-key different-payload requests.
- Target:
  - API returns `409 Conflict` and does not issue a new address.
